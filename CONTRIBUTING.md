# Contributing to MenYou

Thanks for taking a look! MenYou is a small project but happy to take PRs.

## Building

```powershell
# .NET 10 SDK is required (the project targets net10.0-windows).
dotnet build src/MenYou/MenYou.csproj
```

The native input bridge (`src/MenYou.Bridge`) is compiled by
`build-bridge.ps1` as a pre-build step. If MSVC isn't installed the script
exits 0 and MenYou falls back to the managed WinEvent monitor at runtime —
you can develop without VS C++ tools.

## Running locally

```powershell
src/MenYou/bin/Debug/net10.0-windows/MenYou.exe
```

Press **Shift+Win** to open the menu. Right-click the tray icon for
Settings.

## Project layout

- `src/MenYou/` — Avalonia app (this is the only managed project).
- `src/MenYou.Bridge/` — native C++ DLL for global input.
- `docs/` — architecture overview + the automation / signing / deployment guide (`AUTOMATION.md`).
- `samples/custom-themes/` — runnable AXAML examples for Settings → Custom.
  Self-contained files (no `x:Class`, no compiled bindings) the user can
  Load… from the dialog.
- `installer/` — Scoop manifest, Chocolatey nuspec; winget is auto-
  generated by the release workflow.
- `.github/workflows/` — CI (build) and release (publish + sign +
  package-manager push).

## Coding style

- File-scoped namespaces, `var` for apparent types.
- Comments explain *why*, not *what*. Pin tricky Windows-API behavior to
  Microsoft Docs / community findings in a comment — future contributors
  shouldn't have to re-derive the same workaround.
- AXAML resources go in `Styles/`, view-side hooks in `Views/<Window>.axaml.cs`,
  business logic in `ViewModels/` or `Services/`.

## Localization

User-facing strings flow through `Platform/Windows/Strings.cs`:

1. The first option is a Windows shell-resource indirect string
   (`SHLoadIndirectString` on `@dllname,-resId`). If the shell ships the
   label, MenYou matches the OS phrasing in every display language for
   free.
2. The fallback is a Jeek.Avalonia.Localization JSON entry under
   `src/MenYou/Languages/`. There are 21 languages today — add your key
   to `en.json` and `pl.json` at minimum; the release process keeps the
   others mechanically translated.

## Pull requests

- Branch off `main`, target `main`.
- **Use Conventional Commits in the PR title.** The PR title becomes
  the squash-merge commit subject, which release-please reads to
  bump the version and regenerate `CHANGELOG.md`. Prefixes:
  - `feat: ...` → Features (minor bump)
  - `fix: ...` → Bug Fixes (patch bump)
  - `perf: ...` / `deps: ...` → patch bump
  - `docs: ...` / `chore: ...` / `refactor: ...` / `ci: ...` →
    hidden from changelog, no release
  - Append `!` for breaking changes (`feat!: ...`).
- Don't edit `CHANGELOG.md` directly — release-please regenerates it.
- The build / CodeQL / Security / CodeRabbit checks all have to pass
  before merge.

## Review + merge cadence

- Every PR gets a CodeRabbit review automatically (configured in
  `.coderabbit.yaml`). Its `APPROVED` review counts toward the
  approval gate.
- Once approved by a code owner, repo collaborator, or CodeRabbit,
  PRs auto-merge after a **7-day soak** measured from approval time
  (workflow: `.github/workflows/auto-merge.yml`). The soak gives a
  one-week window to catch late-breaking concerns.
- **Dependabot PRs skip the soak** and merge as soon as CI is green.
- Owner-driven `chore` / `ci` / `docs` / `refactor` / `build` PRs are
  auto-approved by `auto-approve-chore.yml`, so they only wait on CI
  and on auto-merge's eligibility check (which for owner PRs requires
  either CodeRabbit approval or a 7-day window from open — see
  `auto-merge.yml`).

## Releasing (maintainers only)

The release pipeline is mostly automatic:

1. Merge PRs to `main` using Conventional Commits in their titles.
2. release-please (`.github/workflows/release-please.yml`) reads the
   merged commits and opens a **"chore(main): release X.Y.Z"** PR
   that bumps the version + writes the next CHANGELOG section.
3. That release PR is auto-merged once CI is green.
4. Merging the release PR pushes the tag `vX.Y.Z`, which triggers
   `release.yml` — publishes the GitHub Release, signs the installer
   (when SignPath is configured), and pushes to winget / Scoop /
   Chocolatey.

To cut a release **out of band**, push a tag manually:

```powershell
git tag -a v0.2.0 -m "v0.2.0"
git push origin v0.2.0
```

`release.yml` still picks that up directly. See
[`docs/AUTOMATION.md`](docs/AUTOMATION.md) for the full picture
(pipeline, code-signing options, secrets to configure once).
